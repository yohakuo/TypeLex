# TypeLex 优化建议与执行计划

## 1. 项目现状

### 1.1 技术栈
- 前端：Next.js 15 + React 19 + TypeScript（`strict`）
- 数据：`localStorage` 本地优先 + 可选 Supabase 快照同步
- 测试：Vitest（以业务逻辑单元测试为主）
- 代码规范：ESLint（`next/core-web-vitals`）

### 1.2 完成度评估
- 当前状态：可用的 MVP+，完成度约 `75% ~ 85%`
- 已完成核心链路：
  - 单词本与单词的增删改查
  - CSV/TXT 导入与去重
  - 按章节/分组学习流程
  - 复习调度与错词收集
  - 本地持久化与可选云同步

### 1.3 已验证结论
- 当前环境下 `next lint` 可通过。
- 缺少 UI/组件层自动化测试。
- 仓库缺少 CI 工作流。

## 2. 关键风险与短板

| 严重级别 | 问题 | 证据 | 影响 |
| --- | --- | --- | --- |
| 高 | 缺少 CI 流程 | 仓库中无 `.github/workflows` | 回归问题可能直接进入主分支 |
| 高 | 测试覆盖偏逻辑层 | Vitest 仅 Node 环境配置 | UI 回归与页面交互问题难提前发现 |
| 高 | Provider 职责过重 | `providers/app-data-provider.tsx` 混合存储、同步、业务逻辑 | 维护成本高，改动风险大 |
| 中 | 导入流程存在 `any` | `app/books/new/page.tsx`、`app/books/[bookId]/page.tsx` | 类型安全在用户输入路径失效 |
| 中 | 发音依赖缺少治理 | `lib/speech/use-speech-synthesis.ts` 直连外部 API | 网络波动下体验不稳定 |
| 中 | 锁文件并存 | 同时存在 `package-lock.json` 与 `pnpm-lock.yaml` | 依赖解析结果不一致 |
| 低 | README 路径不一致 | 文档写 `D:\Projects\TypeLex`，实际为 `D:\Aproject\TypeLex` | 新同学上手成本增加 |

## 3. 优先级优化路线图

### P0（立即执行，稳定性优先）

| 行动项 | 原因 | 预期收益 | 验收标准 |
| --- | --- | --- | --- |
| 建立 CI：`lint + test + build` | 缺少基础质量门禁 | 防止明显回归进入主分支 | PR 自动执行并在失败时阻断合并 |
| 增加 UI/组件测试（`jsdom`） | 仅逻辑测试无法覆盖页面行为 | 更早发现关键流程断裂 | 核心页面具备稳定渲染与交互测试 |
| 清理导入流程 `any` | 用户输入路径应保持强类型 | 提升重构安全性、减少隐性运行时错误 | `rg -n "\bany\b"` 不再命中上述页面 |

### P1（近期执行，可维护性）

| 行动项 | 原因 | 预期收益 | 验收标准 |
| --- | --- | --- | --- |
| 按职责拆分 Provider | 当前 Provider 体量过大 | 降低理解成本，减少改动耦合 | 存储/同步/领域逻辑拆分为独立模块 |
| 增加快照 Schema 校验与迁移 | 当前解析对数据结构信任过高 | 提升数据演进稳定性 | 历史或损坏快照有可预测处理路径 |
| 强化发音请求策略（超时/重试/缓存） | 外部 API 非确定性 | 提升弱网和离线场景体验 | 有明确降级策略与可观测失败行为 |

### P2（中长期执行，治理完善）

| 行动项 | 原因 | 预期收益 | 验收标准 |
| --- | --- | --- | --- |
| 统一包管理器并移除另一份锁文件 | 降低环境差异与安装歧义 | 团队依赖解析一致 | 仓库仅保留一个锁文件并写入贡献规范 |
| 校准 README 实际安装路径与步骤 | 降低上手摩擦 | 新成员可快速完成启动 | 全新环境按 README 可一次跑通 |
| 补充同步冲突/离线行为说明 | 产品行为复杂，需明确预期 | 降低支持成本和误用 | 用户可见文档覆盖冲突与离线场景 |

## 4. 分阶段执行计划

### 本周
- 建立 CI 流程：`lint`、`test`、`build`。
- 清理导入页面中的 `any`，改为明确输入类型。
- 修正 README 路径与环境配置说明。

### 两周内
- 建立 `jsdom` 测试轨道，覆盖关键页面与核心交互。
- 启动 Provider 模块化（优先拆同步与存储能力）。
- 落地发音请求治理策略（超时 + 重试 + 缓存）。

### 长期
- 建立数据快照版本迁移机制。
- 完善同步冲突处理与离线策略文档。
- 在团队协作规范中固化包管理器策略。

## 5. 验收标准

- 质量门禁：
  - 每个 PR 自动执行 CI，并在失败时阻止合并。
  - 在干净环境下可稳定通过 `lint` 与 `build`。
- 测试覆盖：
  - 现有业务单元测试持续通过。
  - 核心页面交互具备组件/集成测试覆盖。
- 架构质量：
  - App Provider 按职责拆分，模块边界清晰。
- 数据可靠性：
  - 持久化数据具备 Schema 校验和版本迁移能力。
- 开发体验：
  - 安装文档与依赖工作流清晰且可重复。
